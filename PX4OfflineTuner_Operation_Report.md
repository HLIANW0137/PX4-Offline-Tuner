# PX4 Offline Tuner 超详细使用与参数操作报告

## 1. 报告目的

本报告基于当前仓库中的 `PX4 Offline Tuner` 软件实现、界面逻辑、推荐算法和已有输出样例整理，目标是回答四类问题：

1. 这个软件到底是干什么的。
2. 这个软件应该怎么用，界面和命令行分别怎么操作。
3. 软件里出现的各种参数、评分、指标、档位分别是什么意思，应该怎么选。
4. 为什么同一次分析中，`roll / pitch / yaw` 三个轴可能会被推荐成不同档位；为什么不是只给三种固定档位，然后每个档位都配一组三轴参数。

这份报告不是泛泛而谈，而是直接结合软件当前行为来说明，所以你拿着它对照界面、命令行和导出报告，都能对得上。

---

## 2. 软件定位

`PX4 Offline Tuner` 是一个面向 PX4 飞行日志的离线自动调参工具。它不是飞控固件，也不是在线实时调参器，而是把历史飞行日志拿进来做离线分析，然后给出一组或多组 PID 推荐值，再通过内部闭环仿真做排序和风险提示。

软件当前覆盖的主要能力包括：

- 读取 `ULog` 或标准化 `CSV`
- 按 `roll / pitch / yaw` 三轴分别抽取速率环数据
- 对日志做重采样、滤波、误差构造和质量评分
- 做频域分析，估计带宽、噪声频率、延迟
- 做简化系统辨识，得到近似模型
- 生成三档 PID 推荐：
  `conservative / balanced / aggressive`
- 用闭环仿真比较三档推荐的表现
- 输出 `report.md`、`report.json`、参数导出文本、每轴图表

一句话理解：

这个软件做的不是“帮你直接把飞机调到最终最优”，而是“利用已有日志，尽量把三轴速率环的初始参数、风险判断和分析报告自动算出来”，用于后续仿真验证、地面验证和实飞前准备。

---

## 3. 软件工作流程总览

你可以把软件理解成一条固定流水线：

1. 读入日志
2. 拆成 `roll / pitch / yaw`
3. 对每个轴分别做预处理
4. 对每个轴分别做频域分析
5. 对每个轴分别做系统辨识
6. 对每个轴分别生成三档 PID 参数
7. 对每个轴分别仿真三档参数
8. 对每个轴分别给出“最佳档位”
9. 最后支持统一导出某一档的三轴参数

这里最关键的一点就是：

**三轴在软件内部本来就是分别分析、分别建模、分别评分、分别排序的。**

这句话几乎就是理解后面所有“为什么三个轴会出现不同推荐”的核心。

## 4. 软件的两种使用方式

当前软件有两种主入口：

- 图形界面：适合大多数用户
- 命令行：适合批处理、自动化或保存固定流程

### 4.1 图形界面启动

命令：

```bash
px4-tuner ui
```

如果没有装入口命令，也可以直接跑 Streamlit：

```bash
streamlit run src/px4_offline_tuner/webapp.py
```

图形界面适合做这些事：

- 选样例日志或上传真实日志
- 设置重采样频率
- 发起分析
- 看总览结论
- 看每轴详细图表和推荐原因
- 导出 `PX4` 参数文本、Markdown 报告、JSON 报告

### 4.2 命令行启动

单日志分析：

```bash
px4-tuner run --input sample_data/demo_log.csv --output outputs/demo
```

多日志联合分析：

```bash
px4-tuner run --input sample_data/demo_log.csv sample_data/another_log.csv --output outputs/multi_demo
```

带重采样频率：

```bash
px4-tuner run --input sample_data/demo_log.csv --output outputs/demo --rate-hz 120
```

命令行适合做这些事：

- 固定流程重复跑
- 做批量日志分析
- 生成标准输出目录
- 适配后续脚本处理

## 5. 输入数据要求

### 5.1 支持的输入格式

软件支持两类输入：

- `.ulg`
- `.csv`

### 5.2 CSV 必填字段

CSV 至少需要这些列：

- `timestamp_s`
- `axis`
- `rate_setpoint`
- `rate`
- `control_output`

可选字段：

- `attitude_setpoint`
- `attitude`
- `battery_voltage`

其中 `axis` 必须是：

- `roll`
- `pitch`
- `yaw`

### 5.3 ULog 的自动映射来源

软件优先从这些 PX4 话题里提取数据：

- `vehicle_angular_velocity`
- `vehicle_rates_setpoint`
- `actuator_controls_0`
- `actuator_motors`
- `actuator_servos`
- `vehicle_attitude`
- `vehicle_attitude_setpoint`
- `battery_status`

如果日志里缺少关键话题，软件会报错或跳过无法构造的轴。

### 5.4 真实使用建议

想让分析更可信，日志最好满足：

- 有明显的速率设定变化，而不是全程几乎不动
- 有足够长的持续时间，不要只有零点几秒
- 控制输出不能永远很小
- 不要全程大面积饱和
- 噪声不要极重

如果日志激励不足，软件还能算，但推荐会明显更偏保守，且报告里会明确写出“辨识信心有限”。

## 6. 图形界面逐区详细说明

软件图形界面主要由侧边栏和 5 个结果页签构成。

### 6.1 侧边栏：运行设置

你会看到以下输入项。

#### 1. 日志来源

可选：

- `样例日志`
- `上传日志`

作用：

- 样例日志适合先熟悉软件
- 上传日志适合分析你自己的飞行数据

选择建议：

- 第一次使用软件，先用样例日志跑一次
- 确认理解输出后，再上传真实日志

#### 2. 选择样例 / 上传文件

作用：

- 指定一个或多个输入日志

选择建议：

- 单日志：用于看单次飞行的响应特点
- 多日志：用于减小偶然性，提高统计稳定性

多日志适合这些场景：

- 同机型多次飞行
- 同一组参数下多段不同动作
- 想看某一轴在多次任务中的平均表现

#### 3. 重采样频率 `target_rate_hz`

这是软件里最重要的操作参数之一。

作用：

- 把原始日志重新插值到固定采样率
- 作为后续频域分析、滤波、仿真采样基础

界面默认值：

- `120 Hz`

命令行默认值：

- `200 Hz`

软件建议区间：

- 大日志或普通分析：`80 ~ 150 Hz`
- 希望更细的时间分辨率：`150 ~ 200 Hz`
- 不建议盲目拉到非常高，除非原始日志质量高且你明确知道自己要看什么

选择逻辑：

- `80 ~ 100 Hz`
  适合大文件、先快速看结论、减少界面卡顿
- `120 Hz`
  适合大部分常规离线分析，是当前界面默认值，属于平衡型选择
- `150 ~ 200 Hz`
  更保留高频细节，但噪声也更容易保留下来，计算和页面负担更高

选低了会怎样：

- 高频细节被抹掉
- 带宽估计可能偏低
- 快速动态可能被看得太“钝”

选高了会怎样：

- 对噪声更敏感
- 页面更卡
- 频谱里高频成分更多，可能让微分项看起来更危险

实战建议：

- 第一次跑真实日志，先用 `120 Hz`
- 日志特别大，先用 `100 Hz`
- 想复核高频噪声或延迟细节，再试 `150 Hz`

#### 4. 输出目录名 `output_name`

作用：

- 决定本次分析结果保存到哪个目录

建议：

- 目录名带日期、机型、测试场景，例如：
  `20260413_quad_test1`
- 不要总是覆盖同一个结果目录，方便你回看不同日志的变化

#### 5. 开始分析

点击后软件会：

1. 读取日志
2. 自动按轴分析
3. 持久化输出结果
4. 刷新显示分析报告

### 6.2 Overview 总览页

这个页签适合做“先看结论”。

主要内容：

- 已分析轴数量
- 平均质量评分
- 平均模型拟合
- 仿真稳定比例
- 每个轴的最佳档位
- 每个轴的超调、上升时间等摘要

你应该怎么读它：

1. 先看三轴是否都被识别出来
2. 再看 `quality_score`
3. 再看 `fit_score`
4. 最后看每轴 `best_preset`

如果一个轴 `quality` 很低或 `fit` 很低，那么它的推荐不能直接当成强结论，更应该当成“参考起点”。

每个轴会单独展示：

- `质量评分`

这页适合回答这些问题：

### 6.4 Exports 导出页

这个页签提供三种导出：

- `PX4 参数`
- `Markdown 报告`
- `JSON 报告`

这里非常关键，因为它体现了软件“实际落地”的方式。

你可以在导出前选择一个档位：

- `conservative`
- `balanced`
- `aggressive`

然后软件会输出这个档位对应的三轴参数：

- `MC_ROLLRATE_P / I / D`
- `MC_PITCHRATE_P / I / D`
- `MC_YAWRATE_P / I / D`

这说明软件其实已经支持了你说的那种模式：

**不是只给单轴看结论，而是可以统一选择一个档位，然后导出这一档下对应的三轴参数。**

所以从落地角度说，软件确实有：

- 三种统一档位
- 每种档位下三轴各自的一组参数

只是与此同时，软件也会在总览里告诉你：

- 从“单轴独立仿真评分”角度看，哪个轴更适合哪一档

这两者并不矛盾，而是两个不同视角。

## 7. 命令行参数详解

### 7.1 `--input`

作用：

- 指定一个或多个日志路径

用法：

```bash
px4-tuner run --input log1.csv
px4-tuner run --input log1.csv log2.csv log3.ulg
```

选择建议：

- 单文件：定位某一次飞行行为
- 多文件：求平均、降偶然性、提升稳定性

### 7.2 `--output`

作用：

- 指定结果输出目录

例如：

```bash
px4-tuner run --input log.csv --output outputs/test_a
```

建议：

- 每次分析单独给目录
- 不要把不同实验混在同一目录

### 7.3 `--rate-hz`

作用：

- 指定分析用重采样频率

例如：

```bash
px4-tuner run --input log.csv --output outputs/test_a --rate-hz 120
```

推荐：

- 初次分析：`120`
- 高质量中小日志：`150 ~ 200`
- 大日志快速分析：`80 ~ 100`

### 7.4 `ui --port`

作用：

- 指定图形界面端口

例如：

```bash
px4-tuner ui --port 8501
```

适合：

- 本机已有其他服务占端口
- 想固定浏览器访问地址

## 9. 软件内部关键指标解释

这一章是理解推荐结果的核心。

### 9.1 质量评分 `quality_score`

这是日志本身值不值得拿来调参的评分。

初始分数是 `1.0`，然后根据问题逐项扣分。

#### 影响质量评分的主要因素

#### 1. 激励强度 `excitation`

本质上看设定变化够不够明显。

如果太弱：

- 扣分
- 说明日志对辨识帮助有限

为什么重要：

- 没有足够激励，系统动态就看不出来
- 参数推荐容易失真

#### 2. 控制输出跨度 `command_span`

看控制输出变化范围够不够大。

如果范围太窄：

- 扣分
- 说明日志没有充分激发控制链

#### 3. 持续时间 `duration_s`

看有效数据持续时长够不够。

如果太短：

- 扣分
- 结果更容易被噪声和偶然波动主导

#### 4. 跟踪误差噪声 `noise`

如果误差非常抖：

- 扣分
- 软件会提醒微分项要谨慎

#### 5. 饱和比例 `saturation_ratio`

看控制输出是否经常接近极限。

如果饱和明显：

- 扣分
- 说明日志里的控制状态不健康
- 激进推荐可能难以迁移到真实飞行

#### 6. 测量速率变化 `tracking_std`

看测得的速率有没有实际波动。

如果波动太小：

- 扣分
- 表示动态信息不足

### 9.2 频域指标 `frequency`

#### 1. 主频 `dominant_frequency_hz`

表示该轴响应中最明显的频率成分。

作用：

- 帮你感知系统主要活动频段

#### 2. 噪声频率 `noise_frequency_hz`

从跟踪误差频谱中找主峰。

作用：

- 判断高频误差是否明显
- 决定微分项是否需要软化

#### 3. 带宽估计 `bandwidth_estimate_hz`

软件按累计功率到 `90%` 的位置估计带宽。

作用：

- 带宽低，通常说明回路偏慢、偏保守
- 带宽高，通常说明回路响应更快，但可能更接近极限

#### 4. RMS 误差 `rms_error`

表示平均跟踪误差大小。

作用：

- 太高说明设定跟踪不佳
- 软件会倾向建议增加控制权威

#### 5. 延迟 `latency_s`

通过设定和响应相关性估算。

作用：

- 延迟大时，激进调参更容易丢鲁棒性
- 软件会自动下调部分 `P` 和 `D`

### 9.3 模型辨识结果 `model`

软件使用的是简化的 `FOPDT` 近似模型，也就是一阶惯性加纯滞后。

#### 1. 增益 `gain`

描述输入控制量对输出速率的放大关系。

影响：

- 增益越大，同样控制更容易产生响应
- 计算出的 `Kp` 往往会相应变小

#### 2. 时间常数 `time_constant_s`

描述系统本身“慢不慢”。

影响：

- 时间常数越大，系统越慢
- 为了把它拉快，推荐参数往往会变大

#### 3. 纯延迟 `dead_time_s`

描述控制作用传到输出前的等待时间。

影响：

- 延迟大时，回路更容易不稳
- 软件会更保守对待 `P` 和 `D`

#### 4. 拟合分数 `fit_score`

描述简化模型和日志真实行为贴合得好不好。

影响：

- 高：说明这个模型比较可信
- 低：说明推荐只能做参考起点

经验上可以这样理解：

- `> 0.75`：可信度较高
- `0.45 ~ 0.75`：可参考，但要结合诊断看
- `< 0.45`：明显偏弱，应优先保守使用

#### 5. 阻尼比 `damping_ratio`

帮助判断系统是否偏轻阻尼。

影响：

- 如果阻尼偏低，软件会略微提高阻尼倾向，通常体现为略调低 `P`、略调高 `D`

## 10. 三档 PID 推荐到底是什么意思

软件当前固定生成三档推荐：

- `conservative`
- `balanced`
- `aggressive`

它们不是三套完全不同算法，而是同一套基础整定公式上的三种闭环快慢目标。

### 10.1 三档的本质区别

软件内部给三档设置了不同的 `lambda factor`：

- `conservative = 3.5`
- `balanced = 2.5`
- `aggressive = 1.7`

你可以把它理解成：

- 数值越大，越保守，目标闭环越慢，鲁棒性优先
- 数值越小，越激进，目标闭环越快，响应速度优先

### 10.2 三档的一般特征

#### `conservative`

特点：

- 风险最低
- 响应最慢
- 通常最适合作为第一次上机或低信心日志的起点

适合：

- 日志质量一般
- 模型拟合偏低
- 延迟偏大
- 噪声偏大
- 你想先求稳

#### `balanced`

特点：

- 速度和鲁棒性的折中
- 最适合作为默认首选档位

适合：

- 大部分普通离线分析
- 希望先拿一组比较均衡的起始参数

#### `aggressive`

特点：

- 追求更快响应
- 风险更高
- 更依赖日志质量和模型可信度

适合：

- 轴向模型清晰
- 延迟不算太糟
- 你明确在追求更快速率跟踪

## 11. Kp / Ki / Kd 在这个软件里怎么被修正

软件先根据辨识模型生成一个基础值，然后再根据日志特征做修正。

### 11.1 噪声高时

如果发现：

- `noise_frequency_hz > bandwidth_estimate_hz * 1.5`

软件会：

- 把 `Kd` 乘以 `0.8`

含义：

- 高频噪声明显时，微分项要软一点
- 否则高频抖动会被放大

### 11.2 延迟大时

如果：

- `latency_s > 0.05`

软件会：

- `Kp` 乘以 `0.9`
- `Kd` 乘以 `0.9`

含义：

- 延迟大的系统不能太猛
- 特别是 `P` 和 `D` 太激进时更容易丢稳态裕度

### 11.3 阻尼偏低时

如果：

- `damping_ratio < 0.45`

软件会：

- `Kp` 乘以 `0.92`
- `Kd` 乘以 `1.05`

含义：

- 稍微弱化冲得太猛的趋势
- 稍微增强阻尼倾向

### 11.4 拟合分数偏低时

如果：

- `fit_score < 0.60`

软件会：

- `Kp` 略降
- `Ki` 略降

如果更低，并且当前是 `aggressive` 档：

- 还会进一步压住激进参数

含义：

- 模型都不太可信了，就不能太相信激进参数

### 11.5 档位本身还会做最后修正

- `conservative` 会把 `Ki` 再压一点
- `aggressive` 会把 `Kp` 和 `Ki` 再推一点

这说明三档的区别不仅体现在基础闭环目标上，也体现在最后的风格化修正上。

## 12. 软件如何决定“最佳档位”

很多人会误以为软件只看推荐公式，其实不是。

真正决定 `best_preset` 的不是公式本身，而是：

**三档参数都先过一遍闭环仿真，再用代价函数排序。**

### 12.1 仿真看哪些指标

每个档位都会算：

- `rise_time_s`
- `settling_time_s`
- `overshoot_pct`
- `steady_state_error`
- `control_effort_rms`
- `peak_control`
- `stable`
- `performance_score`

### 12.2 排序代价看哪些项

软件内部会对这些项加惩罚：

- 不稳定：重罚
- 上升时间慢：罚
- 调节时间慢：罚
- 超调大：罚
- 稳态误差大：罚
- 控制量 RMS 大：罚
- 峰值控制超限：罚

最后代价最小的那个档位，就是该轴的 `best_preset`。

所以：

**最佳档位不是拍脑袋写死的，而是每个轴自己经过仿真排序算出来的。**

## 13. 为什么三个轴还能推荐不同档位

这是你最关心的问题，这里单独展开。

### 13.1 根本原因：三个轴不是同一个系统

即使是同一架飞机，`roll / pitch / yaw` 的动力学也不一样。

原因通常包括：

- 机体几何结构不同
- 转动惯量不同
- 螺旋桨和电机布局对各轴作用不同
- 机架刚度不同
- 气动耦合不同
- `yaw` 通常天生和 `roll/pitch` 不同
- 日志中的激励强弱也可能不同

所以三轴天然就可能表现出：

- 不同带宽
- 不同延迟
- 不同噪声水平
- 不同可辨识性
- 不同拟合分数

既然输入特征不同，最后最优档位不同就非常正常。

### 13.2 软件层面是如何造成这种“分轴不同”的

软件做的是下面这件事：

1. 每个轴单独构造 `AxisDataset`
2. 每个轴单独做频域分析
3. 每个轴单独做 FOPDT 辨识
4. 每个轴单独生成三档参数
5. 每个轴单独跑三档仿真
6. 每个轴单独选代价最低的档位

因此：

- `roll` 可能最佳是 `aggressive`
- `pitch` 可能最佳是 `conservative`
- `yaw` 可能最佳是 `balanced`

这是软件设计上刻意允许的结果，不是异常。

### 13.3 为什么不是“只有三种档位，然后每一档都给三轴参数”就完了

严格说，软件其实已经支持这个模式了。

在导出页里，你选择某个档位后，软件会导出该档位下三轴对应的一组参数，例如：

- 选 `balanced`
- 导出 `roll balanced` 的 P/I/D
- 导出 `pitch balanced` 的 P/I/D
- 导出 `yaw balanced` 的 P/I/D

也就是说：

**软件同时具备这两种能力：**

1. 按轴告诉你“各轴独立最优档位”
2. 按统一档位导出“三轴一整套参数”

这两种能力服务的是不同目标。

#### 视角 A：按轴独立最优

用途：

- 诊断每个轴状态
- 看哪个轴更保守、哪个轴更激进
- 帮你理解三轴动态差异

优点：

- 信息量大
- 能发现单轴异常
- 更符合“每个轴动力学不同”的事实

缺点：

- 真正落地时，你要决定是否接受“三轴档位不一致”

#### 视角 B：统一选择一个档位导出三轴参数

用途：

- 实际给 PX4 导参数
- 形成一套整机起始参数

优点：

- 更整齐
- 更好执行
- 更适合作为第一轮落地参数

缺点：

- 会牺牲部分单轴最优性

所以不是“软件没做三种档位对应三轴参数”，而是：

**软件既做了统一档位导出，也保留了单轴独立最优的诊断结果。**

### 13.4 哪种更适合真实使用

如果你问“实际落地时应该怎么选”，建议这样分层：

#### 场景 1：第一次上机、先求稳

优先选择：

- 统一导出 `balanced`
- 如果某个轴风险很高，再考虑统一降到 `conservative`

原因：

- 好执行
- 风险可控
- 不会因为单轴过激进把整体风险拉高

#### 场景 2：看见一个轴明显更差

例如：

- `pitch` 被判成 `conservative`
- `roll / yaw` 被判成 `aggressive`

这往往意味着：

- `pitch` 轴模型更差、延迟更大、或者噪声更重

这时更合理的操作往往不是“全机都 aggressive”，而是：

- 先统一用 `balanced`
- 重点盯住表现差的那个轴

#### 场景 3：你允许三轴参数不完全同风格

PX4 的速率参数本来就是按轴独立的：

- `MC_ROLLRATE_*`
- `MC_PITCHRATE_*`
- `MC_YAWRATE_*`

从技术上说，混合选择是完全可行的。

例如：

- `roll` 用 `aggressive`
- `pitch` 用 `conservative`
- `yaw` 用 `balanced`

但这样做的前提是：

- 你知道自己在做什么
- 你能接受后续验证复杂度增加
- 你会做额外实飞验证

所以对多数用户，软件虽然允许你看见“分轴不同最佳档位”，但默认落地建议仍然应该是：

**先选统一档位导出，再做逐步验证。**

## 14. 结合现有样例结果解释“三轴不同档位”

当前仓库已有一个样例结果，摘要大意如下：

- `roll` 最佳档位：`aggressive`
- `pitch` 最佳档位：`conservative`
- `yaw` 最佳档位：`aggressive`

这正好是你提的问题的真实例子。

### 14.1 为什么 `pitch` 更保守

样例里 `pitch` 的特征是：

- `fit_score` 很低
- `time_constant_s` 明显更大
- 延迟也不小

这意味着：

- 该轴看起来更慢
- 模型可信度更差
- 软件自然会把它往保守方向拉

所以 `pitch` 被判成 `conservative` 很合理。

### 14.2 为什么 `roll / yaw` 更激进

样例里：

- `roll` 拟合比分 `pitch` 高得多
- `yaw` 的拟合也明显好于 `pitch`

在仿真比较后，这两个轴的激进档位虽然风险更高，但综合代价更低，所以被排成最佳档位。

### 14.3 这个结果说明什么

它不是在说：

- “整机就应该无条件采用 roll aggressive、pitch conservative、yaw aggressive”

而是在说：

- “如果单独看每个轴自己的仿真结果，这三个轴对档位的偏好不一样”

这个差异本身就是非常有价值的信息。

## 15. 实际参数选择建议

如果你要把软件真正用于一轮调参，建议按这个顺序。

### 15.1 第一次跑日志

建议设置：

- 输入：单个代表性日志
- 重采样频率：`120 Hz`
- 先看 `Overview`
- 再逐轴看 `Axis Analysis`

重点确认：

- 三轴是否都识别成功
- `quality_score` 是否过低
- `fit_score` 是否太差
- 有没有明显的“噪声高”“延迟大”“饱和明显”

### 15.2 如果质量一般

建议：

- 优先看 `conservative` 和 `balanced`
- 不要急着用 `aggressive`

### 15.3 如果质量较好、拟合也高

建议：

- 可以重点看 `balanced` 和 `aggressive`
- 但仍要结合时延和噪声判断

### 15.4 如果三轴最佳档位不一致

多数情况下建议：

- 第一轮统一导出 `balanced`

如果某个轴非常差：

- 第一轮统一导出 `conservative`

如果你已经有经验并愿意承担更高验证成本：

- 可以手动做分轴混合

### 15.5 多日志联合分析时怎么选

如果你拿多份日志一起跑，软件会按质量和拟合给权重做加权聚合。

这适合：

- 同机型多次飞行求平均
- 降低一次异常飞行对结论的影响

但前提是：

- 多份日志要大体可比
- 不要把完全不同任务状态、完全不同机况的日志胡乱混在一起

## 17. 常见误区

### 误区 1：最佳档位就是最终实飞参数

不是。

它只是离线推荐结果，还需要后续验证。

### 误区 2：三轴应该永远用同一强度

不一定。

三轴动力学本来就可能不同。

### 误区 3：既然软件给了每轴最佳档位，那就一定要混搭

也不是。

对大部分用户，第一轮更建议统一档位导出。

### 误区 4：日志能跑出结果，说明结果就一定可信

不是。

要先看：

- `quality_score`
- `fit_score`
- 诊断说明

## 18. 最终结论

### 18.1 关于“为什么三个轴还能推荐不同档位”

因为软件是按轴分别做：

- 预处理
- 频域分析
- 系统辨识
- 参数生成
- 仿真排序

三轴的动态特性本来就不同，所以各自最佳档位不同是正常现象。

### 18.2 关于“为什么不是三种档位，然后每种档位有三轴推荐参数”

实际上软件已经这样做了。

在导出页里，你选任意一个档位，软件都会导出这个档位对应的三轴参数。

也就是说，软件同时提供：

- “统一档位下的三轴参数”
- “每轴独立最优档位”的诊断结果

前者用于实际落地，后者用于理解差异和辅助决策。

### 18.3 对多数用户的最佳实践

最稳妥的使用方法是：

1. 先跑日志，看三轴质量和拟合
2. 把每轴最佳档位当作诊断信息
3. 第一轮优先统一导出 `balanced`
4. 若某轴明显偏差，再考虑 `conservative`
5. 等你对三轴差异足够有把握后，再考虑分轴混合参数

## 19. 一页式实操建议

如果你只想要一份最直接的操作建议，可以看这段。

### 推荐默认设置

- 日志来源：先样例，后真实日志
- 输入数量：先单日志，再多日志
- 重采样频率：`120 Hz`
- 第一轮导出档位：`balanced`

### 什么时候改成 `conservative`

- 日志质量一般
- 拟合低
- 延迟大
- 噪声明显
- 某个轴看起来很不稳定

### 什么时候考虑 `aggressive`

- 质量评分高
- 模型拟合高
- 延迟不大
- 你明确要更快响应
- 你愿意承担更高验证成本

### 如何理解“三轴档位不一致”

- 把它看成三轴动态差异的诊断信息
- 不要急着理解成必须混搭落地
- 第一轮仍建议统一档位导出
